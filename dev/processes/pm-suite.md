# PM Suite — project-management layer over the task pipeline

The `task-*` skills are **IC work** (one task: intake → review) and `/task-cowork` is the
**execution orchestrator** that drives them. The `pm-*` skills are the **management brain** above
both: the cross-task, over-time, delivery view. You steer here; `task-cowork` executes what you queue.

```
pm-* layer   — portfolio · over-time · delivery      ← you steer
   │ decides WHAT + queues it
   ▼
task-cowork  — execution orchestrator
   │ runs the stages
   ▼
task-* pipeline — per-task IC work
```

PM skills = **decide + report**. `task-cowork` = **do**. A PM skill never implements a task; it
shapes the backlog and reports, then leaves execution to `task-cowork` / the `task-*` stages.

## Foundation (Phase 0 — enablers)

These primitives are greenfield in this repo today (no milestones, releases, tags, or priority
labels). The PM skills depend on them:

- **Priority labels** — `P0` (critical) · `P1` (high) · `P2` (normal) · `P3` (low). Until now
  priority lived only in `00-task.md` headers; promoting it to GitHub labels makes status/plan/triage
  queryable via `gh`. The pipeline priority model (Critical > High > Normal > Low; type tie-break
  bug > feature > refactoring > research > docs > chore) maps onto these.
- **Milestones** = roadmap buckets (themes / iterations / releases). Issues belong to at most one.
- **Releases & tags** = shipping units. SemVer tags (`v0.1.0` …) + GitHub releases with generated notes.
- **Committed snapshots** (GitHub is source of truth; repo holds a generated mirror for offline reading):
  - `dev/roadmap.md` — milestones in order, with their issues and status. Regenerated by `/pm-roadmap`.
  - `CHANGELOG.md` — release notes. Appended by `/pm-release`.
    Snapshots are **generated, never hand-edited** — edit GitHub, then regenerate.

## Shared conventions (all pm-\* skills)

- **Read the board from**: top-level `dev/tasks/*/` (excluding `archive/`) for stage, plus
  `gh issue`/`gh pr` for status/labels/milestone. Stage = highest `NN-*.md` artifact present.
- **Status taxonomy** (per task): `ready` · `waiting-human` (artifact has an open `## ⏳ NEEDS HUMAN`
  block) · `done` (`05-review.md` exists, awaiting merge) · `shipped` (PR merged → folder archived).
- **Mutations are confirmed**: anything outward-facing (creating/closing issues or milestones,
  pushing labels, cutting a release/tag) is proposed first and run only on explicit go — except
  when invoked unattended via `/loop`, where each skill states its own guardrails.
- **Snapshots stay in sync**: a skill that changes GitHub roadmap/release state regenerates the
  corresponding `dev/roadmap.md` / `CHANGELOG.md` in the same run.

## The skills

| #   | Skill         | Question it answers                  | Reads                               | Writes / acts                                                            |
| --- | ------------- | ------------------------------------ | ----------------------------------- | ------------------------------------------------------------------------ |
| 1   | `/pm-status`  | "Where are we _right now_?"          | board, issues, PRs, quality-history | — (read-only report)                                                     |
| 2   | `/pm-triage`  | "Tidy the backlog."                  | new/unlabeled/stale issues          | issue labels, dedupe, close, split                                       |
| 3   | `/pm-roadmap` | "Where are we going, in what order?" | issues, milestones                  | milestones + `dev/roadmap.md`                                            |
| 4   | `/pm-plan`    | "What do we do _next_?"              | roadmap, board, ready/blocked       | priority/milestone on issues; may call `/task-add`; queues `task-cowork` |
| 5   | `/pm-release` | "Ship it."                           | merged PRs, closed issues, last tag | git tag, gh release, `CHANGELOG.md`, batch-archive done folders          |
| 6   | `/pm-report`  | scheduled _delta_ digest             | board over time (since last report) | digest (issue comment / file)                                            |
| 7   | `/pm-retro`   | "What did we learn?"                 | closed issues, stage dwell times    | `dev/retros/<milestone>.md`                                              |

### 1. /pm-status — on-demand dashboard (read-only)

The steering view. One snapshot: board by stage (00→05) across active tasks; open issues by type ×
priority; open PRs + CI state; `waiting-human`/blocked items **with age**; recently merged/shipped;
quality trend from `dev/quality-history`. Never mutates anything. This is the human's "what's going on".

### 2. /pm-triage — backlog hygiene

Sweeps the **existing** pile (not new intake — that's `/task-add`): set missing type+priority labels,
flag/merge duplicates, close stale `wontfix`/`invalid`, flag too-big issues to split into children.
Proposes a batch of label/close actions; applies on confirm.

### 3. /pm-roadmap — themes over time

Groups issues into milestones and **orders** them. Shows this-iteration vs next vs later, big-item
dependencies, what's next. Source of truth = GitHub milestones; regenerates `dev/roadmap.md`.

### 4. /pm-plan — active planning (the human's main verb)

The bridge from roadmap (strategy) to execution. Take the backlog + roadmap: (re)prioritize, assign
ready tasks to the current milestone/iteration, separate ready from blocked, and **propose the next
batch** for `task-cowork` to run. May spawn `/task-add` for newly surfaced work. Output is a queue,
not code.

### 5. /pm-release — cut a release

Determine shippable scope (merged since last tag), bump SemVer, generate notes from merged PRs +
closed issues, create the git tag + GitHub release, close the milestone, and **batch-archive** the
completed task folders (`git mv dev/tasks/<slug> dev/tasks/archive/<slug>` — see
[README.md](README.md) _Lifecycle_). Never pushes to `main` unattended.

### 6. /pm-report — scheduled delta digest

The `/loop`-driven sibling of `/pm-status`: **since the last report** — shipped, newly blocked,
aging risks, velocity. Status is on-demand & full; report is periodic & diff. Posts the digest.

### 7. /pm-retro — post-milestone review

After a milestone/release closes: cycle time per task, where tasks **stalled** (stage dwell time),
shipped-vs-planned, 2–3 concrete process improvements. Writes `dev/retros/<milestone>.md`.

## Updating the skills

Same rule as the task pipeline: edit the installed copy in `~/.claude/skills/pm-<name>/SKILL.md`
(what Claude Code loads), then mirror here so the process stays versioned with the repo:

```bash
for s in pm-status pm-triage pm-roadmap pm-plan pm-release pm-report pm-retro; do
  cp ~/.claude/skills/$s/SKILL.md dev/processes/$s.md
done
```
